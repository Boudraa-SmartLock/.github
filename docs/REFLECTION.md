## Reflection

This project was my first time building a full-stack system that spans hardware, firmware, multiple backend services, a mobile app, and cloud infrastructure. It pushed me outside of my comfort zone in almost every area, and looking back, most of what I learned came from the mistakes I made along the way.

One of the biggest struggles I had on this project was deploying to Azure. I initially tried to do it the hard way, creating a working ASP.NET API in Visual Studio as a regular project, then manually creating the App Service in the Azure portal and trying to link my app to it after the fact. I ran into issue after issue: environment variables not being picked up, mismatched .NET runtimes, the region I picked being overloaded with free-tier services. It was painful. What I didn't know was that you could just create an Azure App Service and link it directly through Visual Studio from the start, and once I figured that out, the whole process went smoothly. I think the lesson there was that I was relying on my existing knowledge of how to build an ASP.NET API from scratch and just assuming it would translate directly to cloud deployment. It didn't, and if I had taken five minutes to look into the recommended workflow before diving in, I would have saved myself hours.

I had a similar experience with the Arduino IDE. Up until about week 8 of the project, I was using it for all of my firmware development because I just assumed it was the best tool for the job. And on the surface it seemed great: built-in board manager, serial monitor, library manager, one-click upload. But there was no real file management, just tabs. No auto-completion, no proper syntax highlighting, no project structure. I got used to it, but eventually I did a bit of research and found PlatformIO, which is a VS Code extension that does basically everything the Arduino IDE does while giving you all the features of a real code editor. The switch was night and day.

The pattern I kept running into throughout this project was suffering through a problem without stopping to ask myself if there was an easier way. And there usually was. The more I worked on this, the more I realized just how many resources and tools are out there that can make development significantly easier, if you take the time to look for them before committing to a path.

On the technical side, one thing that surprised me was how much complexity comes from getting multiple apps to agree on the same data. The RFID UID hashing is a good example: the ESP32 hashes the UID using mbedtls in C++ when checking it against the whitelist, the mobile app hashes it using expo-crypto in TypeScript when registering a new one, and the DB API stores and compares the hashed values in .NET. If the salt, encoding, or algorithm drifts even slightly between any of those three, nothing matches and key lookups silently fail. Debugging that kind of cross-platform issue is not something I had dealt with before, and it taught me a lot about how important it is to define shared constants and contracts early, and to test across boundaries instead of just within a single codebase.

The other major challenge was feature creep. This is one of the first projects I've built that has real, everyday application. I actually use this thing on my door, and it might genuinely make my life a little easier. Because of that, it was really hard to stop myself from adding more and more features. Some of the things I wanted to build but ultimately couldn't because they were out of scope:

- Push notifications for lock/unlock and door open/close events
- More granular settings (like muting certain event types)
- A welcome/onboarding screen, more polished animations, and better styling overall
- Machine learning anomaly detection, something I was really hoping to have time to implement

What I took away from this is the importance of being realistic about what I can actually deliver. That means honestly evaluating my current knowledge of the subject matter (for push notifications and ML, basically zero), the time I actually have available (the projected 9 hours per week), and whether a feature actually adds meaningful value to the MVP or is just cool to have. Balancing all of those factors is a skill, and going through this project forced me to practice it in a way that I know will carry over into the workplace.

If I were to start this project over, I'd spend the first week just researching tooling and deployment workflows before writing a single line of code. I'd also define the API contracts and shared constants (like the UID salt) upfront in a single document that all three codebases reference. However, resolving all of that the hard way is what made this project valuable. I came out of it with a much better understanding of how real systems fit together, and much more respect for the gap between "it works on my machine" and "it works in production."
